# Delta Architecture

Delta is a native macOS 26 backup manager with two deliberately separate engines. Restic owns Delta-format repository encryption, snapshots, deduplication, restore, retention, pruning, and checks. For Time Machine-format destinations, macOS `backupd`, DiskImages, and APFS own backup and restore semantics while Delta owns the remote object transport, bounded cache, authenticated generations, credentials, coordination, and truthful presentation.

## Process model

| Process | Lifetime and boundary |
| --- | --- |
| `Delta` | User-facing SwiftUI/AppKit process. It edits configuration, coordinates interactive work, observes scheduled work, and hosts the Sparkle updater. |
| `DeltaAgent` | Short-lived Service Management agent stored at `Contents/Resources/DeltaAgent`, matching the bundled launch-agent `BundleProgram`. It wakes at a bounded interval, execs the main Delta executable for due-work evaluation, and exits. |
| `DeltaSecretBridge` | Restricted compatibility executable for noninteractive Keychain access. Normal scheduled password resolution uses the main app executable so Keychain trust follows one signed identity. |
| `DeltaTimeMachineService` | Long-lived Service Management agent only for Time Machine disks being prepared or kept mounted. It owns the bounded cache, authenticated remote generation, local/remote writer locks, App-Group Unix-socket server, lease renewal, and reconnect retry. |
| `DeltaTimeMachineFS` | User-approved, sandboxed FSKit extension. It exposes sparse logical placeholders and forwards regular-file payload operations over an authenticated same-user Unix socket in Delta's macOS App Group to the signed storage service; it does not hold provider credentials, SQLite, or client/server network access. |
| `DeltaTimeMachineHelper` | On-demand privileged Service Management helper stored at `Contents/MacOS/DeltaTimeMachineHelper`, matching the bundled launch-daemon `BundleProgram` and Apple's current executable layout. It is restricted to Delta's exact Team ID and identifiers, validates the already user-mounted Delta filesystem and attached APFS disk, then adds or removes only that exact macOS Time Machine destination. It never receives the disk password and does not mount FSKit, create or attach images, assign APFS roles, or transport backup data. |
| `restic` | Bundled child process with a curated environment. It never receives a destination password in command-line arguments or the ambient process environment. |
| `rclone` | Bundled optional transport used by restic's configured rclone backend and directly by the Time Machine object layer for supported remote providers. |

When an installed update changes either Time Machine Service Management executable or plist, Delta waits until its persisted state proves that no Time Machine disk is connected and no operation is active, then re-registers both components through public `SMAppService` APIs. This follows Service Management's update contract and lets macOS retain authority over approval. Delta waits for asynchronous unregistration to complete, applies a short bounded retry only while macOS still reports the item as unregistered, and then requires an authenticated XPC readiness response whose running code hash exactly matches the embedded helper before recording the installed component fingerprint. One automatic replacement is attempted per fingerprint and app process; a failed proof stops automatic registration churn and exposes one Permissions recovery action. Delta does not edit Background Items or launchd state directly.

The privileged helper is a current `SMAppService` daemon, not an `SMJobBless` tool. Its executable therefore lives in the app's canonical `Contents/MacOS` code directory; release validation rejects the obsolete `Contents/Library/LaunchServices` placement reserved for the deprecated `SMJobBless` layout. The launch-daemon property list alone lives under `Contents/Library/LaunchDaemons`.

The main app and agents share typed `DeltaCore` boundaries. Delta has no kernel extension or listening network service. The FSKit extension is a supported user-space system extension, the storage service runs as the signed-in user, and the privileged helper is on-demand rather than a persistent root daemon.

Scheduled-service registration is gated by a fingerprint of the bundled agent and launch-agent property list. Delta registers an unregistered service, re-registers an enabled service when those artifacts change, and directly registers a service that macOS reports as not found only when the current bundle fingerprint proves both artifacts exist. This repairs stale registrations left by an earlier app layout while keeping a genuinely incomplete bundle fail-closed.

Time Machine system-support registration is likewise bound to a streaming fingerprint of the normalized installed bundle path, user service, privileged helper, property lists, and FSKit extension. Production registration and privileged mutations require the exact `/Applications/Delta.app` installation, so a renamed acceptance copy, DerivedData product, `dist` artifact, or worktree build cannot replace the production Background Items records. Ordinary disconnect remains available from a noncanonical copy so cleanup cannot be stranded. The path witness makes a relocated app detectable even when every embedded byte is unchanged and macOS still retains the earlier bundle location. An idle installed-app update performs the public replacement once, then proves that the exact new helper can launch before the installed fingerprint becomes current. If Service Management still points at an older or moved app, reports an enabled item that cannot run, or launches a stale helper, the proof fails closed and the explicit **Set Up** action re-registers the affected helper and user service through the same public lifecycle. A component newly registered during that request is not torn down merely because its sibling was stale, avoiding a second privileged approval cycle. macOS remains authoritative and may request renewed Login Items approval. Delta never edits launchd or approval databases directly.

Release acceptance uses that same production verifier rather than a test-only launch. On a clean acceptance host, both Time Machine services must begin unregistered, the exact notarized candidate must be installed at `/Applications/Delta.app`, an administrator must approve the launch daemon through System Settings as Apple's `SMAppService` contract requires, and both public registrations must become enabled. Delta then opens the helper's production Mach service and accepts readiness only when the running process returns the Security code hash of the helper embedded in that exact app. The evidence is bound to source commit, version, build, app CDHash, helper CDHash, canonical path, clean initial state, and supported unregister cleanup; production publishing rejects missing, stale, or mismatched evidence. Headless automation cannot substitute direct execution, `launchctl`, private Background Task Management changes, or a synthetic approval for that native administrator boundary.

## Data flow

1. A native file picker records security-scoped bookmarks for selected sources, destinations, and restore targets.
2. Delta stores profile, destination, job, log, issue, and restore-point metadata in SQLite through GRDB.
3. Destination passwords and provider credentials are stored as generic-password items in the login Keychain.
4. Before work begins, Delta validates the profile, resolves source bookmarks, checks destination availability, and acquires a cross-process destination lock.
5. `ResticCommandBuilder` constructs a command and a curated environment. Passwords are supplied through a signed password command or short-lived standard input.
6. `ResticRunner` streams UTF-8-safe stdout/stderr, maps exit states, redacts sensitive values, applies command-specific bounded capture, and persists bounded operational history. Complete structured responses fail closed if their parsing limit is exceeded; long-running operational commands retain their final output tail.
7. Successful backup or cleanup work refreshes the authoritative restore-point cache from restic.

For a Time Machine destination, Delta instead creates an immutable remote recovery bootstrap and starts the user storage service. FSKit modules are approved and discovered per signed-in user, so the main app mounts a private FSKit namespace in that same user session, creates or opens the encrypted sparsebundle through the mounted namespace, attaches it with DiskImages, and assigns its APFS Backup role. Only the resulting APFS volume appears below `/Volumes`. The privileged helper then independently proves the exact FSKit source, sparsebundle, attached `/dev/disk…s…` device, APFS mount, and global Time Machine identity before changing macOS destination configuration.

Public `tmutil` requires both root authorization and Full Disk Access to add or remove a destination. Delta treats that as a system configuration prerequisite, not as storage access: it preflights the installed app's FDA readiness before a first connection can register the verified disk and before explicit removal can deregister it. Ordinary Disconnect never mutates global Time Machine configuration; it detaches DiskImages and FSKit while preserving the exact destination ID, matching the behavior of unplugging a physical backup disk. Explicit Remove is available only while the disk is connected, so the helper can independently match the saved ID to the currently attached verified disk before deregistration. The signed Service Management helper retains the root-only mutation boundary. FSKit, DiskImages, APFS role assignment, the extension-to-service channel, and ordinary disconnect do not require FDA; `backupd` remains the process that reads Time Machine backup sources.

FSKit forwards sparsebundle file operations over an owner-only Unix socket inside the supported `BJCVJ5G7MJ.deltatm` macOS App Group container. The team-prefixed group is a public macOS App Group form and is declared only by the main app, storage service, and FSKit extension; it grants no Internet client or server access. A compact repository-derived socket name stays within Darwin's 103-byte AF_UNIX path limit, while every request still carries the complete repository UUID as the authoritative routing identity. The extension mounts the exact private Application Support source directly, without a temporary alias. Same-user credentials and the kernel-supplied immutable audit token must also match Delta's exact team and component identifiers.

The FSKit readiness probe binds the enabled module URL to the extension inside the currently running app rather than accepting another registered development or stale app copy with the same identifier. The storage service independently binds an FSKit socket peer's dynamic CDHash to the statically signed extension executable inside the same installed app bundle. A stale mapped extension process therefore cannot drive a newly installed storage service merely because its Team ID and identifier still match; it fails closed before any disk request is dispatched.

The outer unary FSKit filesystem is a connection transport, not the backup disk. Each connection therefore uses one generated mount-session UUID as its public FSKit container/volume identity and exact private mount-point suffix. The repository UUID remains stable and is independently validated from private owner-only source markers and every IPC request. The encrypted sparsebundle and its inner APFS Backup volume retain the durable identities that DiskImages and Time Machine observe. This separation also lets a new supported connection coexist with an Apple-owned stale LiveFS record from a previously crashed transport instance; Delta never edits private LiveFS state or restarts a system filesystem daemon to manufacture recovery.

The public FSKit mount uses macOS's `noowners` semantics, so mounted-view ownership is projected as the effective user of the observing process. Delta therefore anchors ownership in the unprojected private source namespace: the source root, repository marker, connection marker, and final sparsebundle directory are validated together through one no-follow descriptor with exact owner, device, type, and owner-only mode requirements. The mounted view is validated independently for the exact session path, FSKit type, reported source, device, directory type, and safe mode. UID equality may be projected only when the kernel explicitly reports `MNT_IGNORE_OWNERSHIP`; all other identity and metadata checks remain mandatory.

Dirty 8 MiB bands begin in a strictly bounded local working window. When that window fills before macOS requests synchronization, the storage service uploads its dirty bands as immutable content-addressed objects, verifies them by read-back under the repository-scoped writer lease, replaces only those verified local payloads with remote references, and retries the exact filesystem mutation. This pre-publication spill does not create a manifest or advance authenticated history: a crash still returns to the preceding signed generation, and the unreachable staged objects are eligible only for authenticated, delayed garbage collection. At a blocking synchronization, Delta verifies every newly referenced object, rechecks lease ownership, and publishes the HMAC-authenticated sharded generation manifest before acknowledging the durability barrier to DiskImages. Spill and final-publication uploads are partitioned into fixed batches of at most 64 MiB, including a large set of changed metadata shards, so transport staging is independent of backup size and user cache configuration. A remote error remains an error on the still-mounted disk; it never becomes a false disconnect or success.

SQLite retains a monotonic local rollback witness containing both the latest accepted generation and its authenticated manifest digest. Service startup, repair, and offline check authenticate the retained manifest suffix, require consecutive generations and exact parent digests after its first retained entry, and prove that the local witness still belongs to that chain. The witness may move forward but never backward or onto a same-generation fork. These checks precede remote lease acquisition and any placeholder, cache, pruning, or garbage-collection mutation; startup rechecks after acquiring the lease before materializing local state. Maintenance deletion is bound to the exact authenticated head and stops if it changes. Runtime history compaction follows, rather than precedes, successful local witness persistence, so a local database failure preserves enough remote history for a safe restart.

Remote verification failures have deliberately different typed presentations. Destination, Dashboard, menu-bar, and notification surfaces use non-technical guidance with one authoritative recovery action; missing or corrupt objects never expose provider keys, content digests, or internal paths there. A missing local or mounted root, rclone command failure, or transport timeout is `Storage Unavailable` with reconnection/access guidance, while an authenticated-history or object failure remains `Verification Failed` with provider-restore guidance. Activity and diagnostics retain the exact redacted technical failure as audit evidence. Both offer `Check Again` after the operator restores the relevant provider condition, rather than routing through setup repair and implying that local configuration can replace remote authority.

The local working-window limit must be at least 64 MiB and strictly smaller than the logical disk. It is a performance and local-footprint control, not a backup-size or source-file-size limit. Clean data may be evicted immediately; dirty data may leave the Mac only after its immutable remote object is uploaded and read-back verified, while the signed generation still waits for macOS's synchronization barrier. Reads, object payloads, manifests, listings, lease records, paths, process output, and rclone staging are independently bounded. The service retains at most two already authenticated 8 MiB bands as a fixed 16 MiB in-memory hot window for DiskImages' repeated small reads. A third miss evicts and overwrites the oldest uniquely owned allocation through the complete local-or-rclone read path, while every socket frame has its own autorelease lifetime; long band scans therefore reuse a constant allocation set instead of leaving one freed large region per band. Remote rclone verification may temporarily allocate one additional fixed 64 MiB transfer batch outside the configured cache; these overheads do not grow with the disk, backup, or largest source file. The visible sparsebundle is reconstructed as metadata and zero-allocation logical placeholders, so a 20 MiB backup and a multi-terabyte backup use the same bounded local mechanism and Delta never needs a complete local image before or after upload.

## Persistence

`~/Library/Application Support/Delta` contains the SQLite database, operational logs, process locks, run-control requests, and strictly bounded reconstructible Time Machine cache/placeholders. This is replaceable application state, not backup data. Encrypted backup data remains at the destination selected by the user. The App Group container holds only private local IPC socket nodes; it is not a cache or sparsebundle copy.

User preferences use Delta's own defaults domain. Secrets use Keychain service `com.delta.backup.destination-secrets`. The app does not write credentials to SQLite, logs, diagnostics, temporary password files, release artifacts, or command arguments.

Provider-credential edits are staged as a recoverable Keychain transaction around the durable destination configuration, validation, and audit-event commit. If that local commit fails, replaced and removed secrets are restored and newly created secrets are deleted. A rollback failure remains an actionable error; Delta does not silently leave the saved configuration and its credential set disagreeing.

Diagnostic reports include the persisted Time Machine lifecycle, committed generation, bounded-cache accounting, failure context, and last error. The same central diagnostic redaction and home-path privacy pass applies to those fields, preserving useful setup-stage evidence without exporting credentials or personal paths.

Time Machine stores publish a fixed-path recovery bootstrap. Its independent random manifest-authentication key is wrapped with AES-256-GCM under a key derived from the encrypted disk password using PBKDF2-HMAC-SHA256 with a per-store salt and 600,000 iterations. Recovery metadata is authenticated as additional data. The plaintext disk password and manifest key remain in Keychain during normal operation; the remote bootstrap contains neither in plaintext.

App-managed Time Machine disk passwords use a Keychain account derived from the immutable remote store ID, not Delta's replaceable local repository ID. Before removing local configuration, Delta loads the active key, refuses a conflicting store-scoped key, copies and reads back an early repository-scoped development key when necessary, and stops before deleting configuration or cache if any retention step fails. Removal may then discard the manifest key and provider credentials while keeping the discoverable disk recovery key for the authenticated Existing Disk workflow.

## Concurrency and failure handling

SQLite uses WAL mode and a busy timeout so the app and scheduled process can coordinate safely. A local file lock serializes Delta work per destination, while restic's repository locks protect against other clients. Persisted running jobs are marked interrupted only after Delta proves that no process still owns the local destination lock.

Operations fail closed on invalid profiles, inaccessible sources, missing credentials, an unavailable destination, a busy repository, or unavailable application state. Restore-to-original-location and in-place overwrite paths require explicit confirmation.

Local and mounted Time Machine roots have an explicit lifecycle policy. First-time preparation may create only the selected final folder beneath an existing parent. Recovery, verification, connection, and the storage service require the configured root to exist; descriptor-relative no-follow traversal reports a missing provider as unavailable instead of treating it as an empty history or recreating the path on a different volume.

Before any FSKit mount, remote write, or global Time Machine mutation, Delta gives the signed helper fifteen seconds to return its current Security code hash over an authenticated XPC connection and requires an exact match to the installed helper. This short readiness boundary prevents a broken Service Management job from consuming the mutation deadline or leaving a partially attached disk. After readiness succeeds, Time Machine connection subprocesses share one monotonic ten-minute operation deadline plus a separately bounded one-minute rollback window, enclosed by the app's twelve-minute privileged-configuration deadline. Concurrent system-destination mutations are rejected. Setup passwords are present only in the user-session disk controller during connect, are passed to DiskImages as mutable short-lived standard input rather than command arguments or environment, and are cleared immediately afterward. Passwords never cross privileged XPC.

Connection, storage, and presentation health have explicit ownership. The storage service may publish cache usage, authenticated generation witnesses, and recovery from its own remote-synchronization failure, but it cannot clear helper, FSKit, DiskImages, `tmutil`, rollback, or persistence failures recorded by the connection coordinator. A disk is backup-ready only when one shared typed invariant contains the exact mount session, sparsebundle path, APFS mount and device, canonical macOS Time Machine destination UUID, and no unresolved failure. A residual mount remains configuration-blocking and safely disconnectable, but Dashboard, Destinations, the menu bar, and the backup command all reject it as a backup target.

Installed Time Machine component fingerprints are computed on the utility-priority system-state path with one MiB streaming reads, then cached on the main actor for update and connection decisions. The normal database refresh loop never rereads whole service, helper, or FSKit executables and never retains the component set in memory at once.

User-disk and helper failures return a structured residual FSKit state (`mounted`, `unmounted`, or `unknown`) observed after rollback. Delta keeps the storage service and mounted lifecycle alive whenever detach is incomplete or cannot be proven; it never tears the backend out from under a live FSKit mount. A confirmed unmount preserves the known global Time Machine destination as the identity for a supported reconnect. Explicit configuration removal requires that disk to be reconnected so the helper can authorize deregistration from attached evidence rather than a saved identifier alone. The per-store writer identity survives a service crash so the restarted service can renew its own orphaned lease without waiting five minutes, while other writers remain excluded. A transport failure likewise preserves the mounted lifecycle until the exact attached disk can be safely detached; it never turns an uncertain mount into a false disconnect.

Outer sparsebundle renames reject self-descendant moves and validate every dirty-cache source and destination without following links. Replacement uses the native atomic swap-rename primitive, so rollback never reads or retains cache-sized destination payloads in memory. Error responses preserve `EINVAL`, `ENOTEMPTY`, `ENOSPC`, lease-busy, network, and underlying POSIX distinctions for FSKit and DiskImages.

Authenticated generation manifests are sharded into individually bounded objects and loaded in bounded transport batches. The optimized runtime head lookup remains separate from startup/offline retained-chain validation, so ordinary filesystem writes do not enumerate historical manifests. Before downloading shards, Delta validates their unique prefixes, encoded sizes, and aggregate declared file count against a limit derived from the maximum supported image capacity. Decoded file and chunk-reference totals are independently bounded, path and logical-byte structure is revalidated, and locally constructed shards must fit the same four MiB object limit.

Persisted lifecycle is not proof of current system state. At launch and on a bounded system-state cadence, Delta read-only reconciles each mounted destination against the exact repository-scoped `delta-tm` mount, expected sparsebundle attachment, APFS device and mount point, the exact device's Backup role from public `diskutil apfs list -plist`, and public `tmutil destinationinfo -X` identity. The user-session controller verifies the role after assigning it, and the privileged helper independently verifies it before registration and before returning success. Only agreement across all layers is presented as Connected. Complete absence becomes disconnected or retryable according to the interrupted operation, a remaining global destination becomes cleanup-required without authorizing deletion from a saved identifier alone, and partial or unobservable state remains cleanup-only and blocks configuration changes.

Scheduled notification submission is part of the short-lived agent lifecycle. After a due job finishes, the command-line process waits for `UNUserNotificationCenter` to acknowledge each eligible request, with a bounded five-second timeout; submission failures and timeouts are recorded as warning events before the agent exits.

## Update and distribution trust

Delta embeds a product-specific Sparkle EdDSA public key and requires a signed feed plus verification before extraction. Release artifacts are universal `arm64` + `x86_64`, signed inside-out with Developer ID, use Hardened Runtime and secure timestamps, and are notarized by Apple. The app archive and DMG are notarized separately; both are stapled and Gatekeeper-assessed before publication.

The release manifest binds public artifact hashes, executable hashes, version/build, architectures, minimum system version, and Git commit. dSYMs and Apple notarization evidence remain private.
